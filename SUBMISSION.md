## What I Built

A buyer-side vehicle auction platform with four core flows.

**Browse** — Searchable inventory grid across 200 vehicles. Search is debounced 300ms and spans make, model, year, trim, and city. All filter state lives in URL query params so filtered views survive navigation and are shareable by URL. Filters combine with AND logic. An Ending Soon strip at the top surfaces live auctions. A stats bar shows total vehicles, live count, ending soon count, and total inventory value.

**Inspect** — Full detail pages with image gallery, specifications, condition section with grade display and damage notes, title status badge, selling dealership, and a Condition Summary card. The Condition Summary is generated by Claude Haiku using the vehicle's actual condition data — condition grade, condition report, damage notes, and title status. All 200 summaries are pre-computed and baked into the dataset so every reviewer sees real AI output with zero latency and no API key required.

**Bid** — Tiered bid increments that model real auction behavior by price tier. Reserve price is never shown — only whether it has been met — following real auction game theory where showing the exact reserve number removes bidding incentive. Buy Now flow available on eligible vehicles. All bid state persists to localStorage and survives page refresh.

**Track** — Watchlist with localStorage persistence, nav badge count, and a dedicated watchlist page.

## Notable Decisions

**URL-persisted filters** — Filter state lives in React Router's useSearchParams rather than component state. Filters survive navigation, the back button restores state, and filtered views are shareable by copying the URL. This is the correct architecture for a buyer who filters to specific criteria and wants to explore vehicles within those criteria.

**Tiered bid increments** — Flat percentage increments don't model real auction behavior. A 5% minimum increment on a $6,000 vehicle is $300 — reasonable. On a $60,000 vehicle it's $3,000 — too aggressive. Real wholesale auctions use price-tiered increments because the meaningful step should be proportional to the asset class. Under $10k: $250, $10k–$25k: $500, $25k–$50k: $750, over $50k: $1,000.

**Reserve price hidden** — Displaying the exact reserve price destroys auction game theory. If a buyer knows the reserve is $24,000 and the current bid is $22,000, they bid $24,001 and stop. Concealing it forces buyers to bid based on what they think the vehicle is worth. This is how every real auction platform works — eBay, Copart, ADESA — none show the exact reserve.

**Pre-computed AI summaries** — Calling the Claude API on every page load adds latency, cost, and a dependency on the reviewer having an API key. The better approach: run the enrichment once at build time, bake the summaries into the dataset, and serve them statically. Every reviewer sees real AI output. Zero latency. No setup required. The scripts/enrich.ts script is idempotent and can be re-run to refresh summaries.

**Starting bid vs current bid** — Zero-bid vehicles show STARTING BID and the starting_bid value rather than CURRENT BID: $0. A $0 current bid implies someone bid zero dollars. Before any bids exist, the relevant number is the floor price.

**Frontend only** — A backend adds a server, a database, environment setup, and more things that can fail when someone clones the repo at 9pm before a review. localStorage persistence is sufficient for a prototype. The bid state does not need to survive across sessions.

## Testing

20 unit tests across 3 files using Vitest.

**bidUtils.test.ts** — 5 tests covering all four tiered increment tiers and boundary values. Confirms that a $9,999 vehicle increments $250, a $10,000 vehicle increments $500, a $25,000 vehicle increments $750, and a $50,000 vehicle increments $1,000.

**formatUtils.test.ts** — 12 tests covering currency formatting, odometer formatting, condition label thresholds (Excellent at 4.0+, Good at 3.0–3.9, Fair at 2.0–2.9, Poor below 2.0), and condition color classes.

**auctionUtils.test.ts** — 3 tests confirming getAuctionStatus returns one of the three valid statuses, is deterministic for the same vehicle ID across multiple calls, and returns different statuses across the dataset rather than collapsing to one value.

All 20 tests pass. Run with npm test.

## AI Workflow

I wrote a full product spec (SPEC.md) and project rules file (CLAUDE.md) before forking the repo or writing any code. Every feature decision was made before prompting — Claude Code executed against a defined spec rather than making product decisions autonomously.

I used Claude Code with Opus for the initial planning session to design the component tree, context shape, and utility function signatures. All implementation used Sonnet. I ran /compact between major features to manage context. I reviewed every component output before accepting it.

During the build I reviewed other candidates' public submissions and updated the spec based on what I observed — this informed decisions around URL-persisted filters and bid increment approach. All three of the decisions documented above (tiered increments, hidden reserve, URL filters) were my calls made before any code was written.

I caught two specific issues during manual browser testing: the bid form was pre-populating with an incorrect amount on zero-bid vehicles, and the placeholder image background was rendering as dark navy from the external image service. Both were caught visually in the browser and corrected.

The AI product feature — the Condition Summary — was designed as a pre-computation rather than a runtime API call specifically to avoid the reviewer dependency problem. The enrichment script (scripts/enrich.ts) runs once, calls Claude Haiku with each vehicle's actual condition data, and writes summaries back to the dataset.

## What I'd Do With More Time

**Compare mode** — Side-by-side comparison of two vehicles with highlighted differences on condition grade, bid amount, and specs. The data structure and compare state in AppContext are already in place. This was scoped out to protect core quality within the timebox.

**Evaluation layer for AI summaries** — A script that runs the summaries through a quality rubric: minimum length, no hallucinated vehicle details, specific damage note references. The enrich script generates summaries but does not validate them systematically.

**Real-time bid updates** — WebSocket connection so bid state updates across tabs and sessions. The current localStorage approach is single-session only.

**Proxy bidding** — Set a maximum bid and let the system auto-increment up to it. More realistic auction mechanic than manual increment bidding.

**Image handling** — Real vehicle photography integration. The current SVG placeholders are clean but obviously placeholder.
